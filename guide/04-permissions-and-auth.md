# 4. Permissions & Authentication

This application uses a role-based access control (RBAC) system built on top of Firebase Authentication and Firestore Security Rules.

## User Roles

There are two primary roles in the system:

1.  **Admin**: Administrators have full, unrestricted access to all data and features in the application. They can create, read, update, and delete anything.
2.  **User**: Standard users have limited access based on the specific permissions granted to them. By default, a new user has no permissions.

## Login Mechanism (`user_lookups`)

To allow users to sign in with a `loginId` or `phone` number instead of just an email, the application uses a clever lookup system.

-   **Collection**: `user_lookups`
-   **How it Works**:
    1.  When a user tries to sign in, the application first queries the `user_lookups` collection.
    2.  It looks for a document whose ID matches the provided `loginId` or `phone` number.
    3.  This lookup document contains the user's actual `email` address.
    4.  The application then uses this retrieved email and the provided password to sign the user in via Firebase Authentication's standard `signInWithEmailAndPassword` method.
-   **Security**: The `user_lookups` collection is publicly readable, which is safe because it only reveals the email associated with a given login ID, not passwords or other sensitive data.

## Permissions Structure

Granular permissions for non-admin users are defined in `src/lib/modules.ts` and stored in the `permissions` field of a user's document in the `users` collection.

-   **Admin Override**: If a user's `role` is set to `Admin`, the `permissions` object is completely ignored. The security rules grant access based on the `isAdmin()` function.
-   **User Permissions**: For users with the `User` role, the application checks the `permissions` object. For example, to check if a user can create a campaign, the code would check `userProfile.permissions.campaigns.create`.
-   **UI Control**: The UI dynamically shows or hides buttons and links based on these permission flags, providing a seamless experience.

## Troubleshooting Access Errors

If you encounter "Missing or insufficient permissions" errors, follow these steps:

1.  **Check the Browser Console**: The developer console will contain a detailed error message generated by `src/firebase/errors.ts`. This message includes the operation (`read`, `write`, etc.) and the Firestore path that was denied.

2.  **Review `firestore.rules`**: Open the `firestore.rules` file and find the `match` block corresponding to the path from the error message.

3.  **Verify the `isAdmin()` Function**: This is the most common point of failure. Ensure your admin user document in Firestore has `role: "Admin"`. The `isAdmin()` function looks like this:

    ```rules
    function isAdmin() {
      return isSignedIn() &&
             exists(/databases/$(database)/documents/users/$(request.auth.uid)) &&
             get(/databases/$(database)/documents/users/$(request.auth.uid)).data.role == 'Admin';
    }
    ```
    This rule checks that the user is signed in, their user document exists, and the `role` field within that document is set to "Admin".

4.  **Check User-Specific Rules**: If the error is for a non-admin user, check the path-based ownership rules. For example, the rule for a user's documents is typically:

    ```rules
    match /users/{userId}/documents/{documentId} {
      allow read, write: if isOwner(userId);
    }
    ```
    Ensure the `userId` in the path matches the authenticated user's UID (`request.auth.uid`).

5.  **Examine `storage.rules`**: If the error is related to file uploads or viewing images, the issue is likely in `storage.rules`. The structure is similar to Firestore, granting admins full access and users access to their own files under `users/{userId}`.
